# 03. Threat Intelligence for SOC

## Introduction

Security Operations Centers need to deal with constantly changing threats such as new malware campaigns, emerging Indicators of Compromise, zero-day vulnerabilities, and unknown adversary activity.

In this room, I focused on understanding how Threat Intelligence can support SOC operations and detection engineering. The main idea was to see how information from reliable sources can be collected, understood, and used to improve prevention, detection, and response capabilities.

Threat Intelligence is important because security teams cannot rely only on internal alerts or default detection rules. Attackers continuously change their infrastructure, tools, and techniques. Because of this, SOC teams need to use external and internal intelligence to stay aware of active threats and prepare better detections.

This room helped me understand that Threat Intelligence is not only about collecting IOCs. It also includes understanding who produces intelligence, who consumes it, what type of intelligence is being used, and how that intelligence can be applied in a real SOC pipeline.

## Threat Intelligence in SOC Operations

Threat Intelligence helps SOC teams make better decisions during monitoring, detection, investigation, and response.

From a SOC analyst point of view, useful threat intelligence can help answer questions such as:

* What threats are currently active?
* Which indicators are related to known malicious activity?
* Which adversary behaviours should be monitored?
* Are there new vulnerabilities that may affect the environment?
* Can this intelligence be turned into detection logic?
* Does this information help improve response actions?

The value of Threat Intelligence depends on how it is used. Simply collecting indicators is not enough. The intelligence should be relevant, reliable, timely, and actionable.

For example, a malicious domain, IP address, file hash, or attacker technique can be used to support SIEM detections, threat hunting, alert triage, or incident response.

This helped me understand that Threat Intelligence becomes more valuable when it is connected to real SOC workflows instead of being treated as isolated information.


## Threat Intelligence Recap

Threat Intelligence is the process of analysing data and information to understand existing or emerging threats that may target an organisation, sector, or environment.

In a SOC environment, threat intelligence is useful because it helps analysts understand what to look for, which indicators may be malicious, and how external information can be used to improve detection and response.

There are different types of Threat Intelligence:

| Type                         | Description                                                                                                                  |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **Strategic Intelligence**   | High-level intelligence used to understand the wider threat landscape, trends, and risks that may affect business decisions. |
| **Technical Intelligence**   | Intelligence based on technical artifacts such as IP addresses, domains, hashes, URLs, malware samples, and other IOCs.      |
| **Tactical Intelligence**    | Intelligence focused on attacker tactics, techniques, and procedures (TTPs).                                                 |
| **Operational Intelligence** | Intelligence related to an adversary’s intent, motivation, campaign, and possible targets.                                   |

In this room, the main focus was **Technical Intelligence**, especially using IOCs to support SOC detection and investigation activities.

From my point of view, this is one of the most practical types of intelligence for SOC analysts because technical indicators can be used directly in SIEM searches, detection rules, blocklists, and investigation workflows.

## Threat Intelligence Consumers and Producers

Threat Intelligence can be created and used by different groups. In this section, I looked at the difference between **Producers** and **Consumers**.

**Threat Intelligence Producers** collect, analyse, and share intelligence. These can include cybersecurity vendors, research teams, internal security teams, government agencies, or organisations that have enough visibility to collect and analyse threat data.

Producers may collect information from:

* Network monitoring
* Endpoint telemetry
* Malware analysis
* Honeypots
* Incident response investigations
* External research
* Internal security events

**Threat Intelligence Consumers** use intelligence created by others to improve their own security posture. SOC teams are often consumers because they use threat intelligence feeds, reports, advisories, and IOCs to support monitoring and detection.

Consumers can use threat intelligence for:

* Identifying vulnerabilities
* Improving prevention and detection
* Supporting incident response
* Enriching alerts
* Validating suspicious activity
* Sharing useful findings with other teams

This helped me understand that an organisation does not always need to produce all intelligence by itself. Even if a SOC team mainly consumes intelligence, it can still create value by validating, applying, and tuning that intelligence for its own environment.

## Using Threat Intelligence in SOC Workflows

Threat Intelligence becomes useful when it is connected to real SOC workflows.

For example, an IOC list by itself is just raw information. It becomes more valuable when the analyst can use it to search logs, detect suspicious connections, enrich alerts, or support an investigation.

In this room, I started working with a set of IP-based IOCs and used them to support hunting in a Kibana environment.

The general workflow was:

* Review the provided IOCs.
* Convert or clean the indicators if needed.
* Use Uncoder to transform the IOC list into a query format.
* Run the query in Kibana Discover.
* Search the correct index and timeframe.
* Review matching events and identify suspicious activity.

This showed me that Threat Intelligence should be actionable. If an IOC cannot be searched, validated, or applied in the environment, its operational value becomes limited.

## Applying Threat Intelligence for Prevention

After using threat intelligence for hunting and investigation, I also looked at how IOCs can be used to deploy preventive security controls.

In a SOC environment, threat intelligence is not only useful for detecting suspicious activity. It can also be used to prevent known malicious activity from reaching the environment or communicating with attacker infrastructure.

The main idea is that if reliable threat intelligence provides known malicious domains, IP addresses, URLs, or other indicators, these indicators can be applied to different security controls.

Common examples include:

* Blocking malicious IP addresses on a firewall
* Blocking malicious domains through an email gateway
* Redirecting malicious domains through a DNS sinkhole
* Using IOCs to improve detection and response workflows

This helped me understand that threat intelligence can support both detection and prevention. A SOC team can consume intelligence from reliable sources and use it to reduce risk before an incident becomes more serious.

## IP Blocking via Firewall

One way to use IP-based threat intelligence is by blocking known malicious IP addresses through firewall rules.

A firewall can be configured to block inbound or outbound traffic based on source or destination IP addresses.

This can help in two main ways:

* **Ingress blocking** can prevent known malicious external IP addresses from reaching public-facing systems.
* **Egress blocking** can prevent compromised internal systems from connecting back to attacker-controlled infrastructure.

From a SOC analyst point of view, this is useful because a known malicious IP address can be turned into a preventive control. However, it is still important to validate the indicator before blocking it, because incorrect blocking may affect legitimate traffic.

## Domain Blocking through Email Gateways

Threat intelligence can also be used to block malicious domains at the email gateway level.

If a domain is known to send phishing emails, spam, or malware delivery messages, an email gateway can block emails from that domain before they reach users’ inboxes.

This is useful because many compromises start with phishing emails, malicious attachments, or links to credential-harvesting websites.

Blocking malicious domains at the email gateway can:

* Reduce phishing exposure
* Prevent malicious emails from reaching users
* Lower the chance of credential theft
* Reduce the burden on user awareness alone

This helped me understand that threat intelligence can be applied before an alert is generated. Instead of only detecting malicious activity after it happens, the organisation can prevent some threats from reaching users in the first place.

## Domain Blocking through DNS Sinkhole

Another prevention method covered in this section was DNS sinkholing.

A DNS sinkhole prevents systems from resolving known malicious domains to their real attacker-controlled IP addresses. Instead, DNS requests for malicious domains can be redirected to a safe internal address or sinkhole destination.

This can be useful when a compromised host tries to communicate with a command-and-control domain.

Without DNS sinkholing, the infected host may resolve the malicious domain and connect to the attacker’s infrastructure.

With DNS sinkholing, the DNS request can be intercepted or redirected, preventing the connection from reaching the real malicious server.

From an analyst point of view, DNS sinkholing is useful because it can support both prevention and detection. If an internal host attempts to resolve a sinkholed malicious domain, that activity may indicate possible compromise and should be investigated.

## Optimising Detection Capabilities

After using threat intelligence for prevention, I moved to the next part of the room, which focused on improving detection capabilities.

The main idea was to use IOCs not only for blocking or prevention, but also for building detection logic that can identify suspicious activity in logs.

From a SOC analyst point of view, this is important because prevention controls are not always enough. Even if a firewall, DNS sinkhole, proxy, or email gateway blocks malicious activity, the related logs can still provide valuable detection opportunities.

For example:

| IOC Type       | Detection Use Case                                                                                                    |
| -------------- | --------------------------------------------------------------------------------------------------------------------- |
| **IP Address** | Firewall logs can show inbound or outbound connections involving known malicious IP addresses.                        |
| **URL**        | Proxy logs can show suspicious HTTP GET or POST requests related to malware delivery, phishing, or data exfiltration. |
| **Domain**     | DNS logs can show internal hosts attempting to resolve suspicious or known malicious domains.                         |

This helped me understand that threat intelligence can be used in different stages of the SOC workflow. It can support prevention, but it can also improve detection and investigation by turning IOCs into searchable logic or alerting rules.

## Sigma Rules Revisited

In this part, I reviewed how Sigma can be used to describe suspicious log activity in a structured and platform-independent format.

The lab used a Sigma rule to hunt for DNS sinkhole activity. The rule focused on DNS queries where the resolved IP address was `0.0.0.0`.

```yaml
title: DNS Sinkhole
author: TryHackMe User
description: Sigma rule for sinkholed DNS queries
logsource:
  category: dns
detection:
  select_sinkholed:
    dns.resolved_ip:
      - '0.0.0.0'
  condition: select_sinkholed
falsepositives:
  - Unknown
status: experimental
level: medium
tags:
  - dns
  - filebeat
```

The purpose of this rule was to detect DNS responses that resolved to `0.0.0.0`. In a DNS sinkhole setup, this kind of response can indicate that a suspicious or malicious domain was blocked or redirected.

> This showed me how a prevention technique such as DNS sinkholing can also create useful detection opportunities for SOC teams.

## Converting Sigma to ElastAlert with Uncoder

After reviewing the Sigma rule, I used **Uncoder.io** to convert the Sigma logic into an ElastAlert rule.

The conversion workflow was:

```text
Sigma → ElastAlert Alert (Lucene)
```

This helped transform the Sigma detection into a format that could be used with ElastAlert and Elasticsearch-based logs.

The generated ElastAlert rule included important fields such as:

| Field       | Purpose                                                         |
| ----------- | --------------------------------------------------------------- |
| **alert**   | Defines the alerting method. In this lab, `debug` was used.     |
| **filter**  | Contains the query logic for detecting sinkholed DNS responses. |
| **index**   | Defines the Elasticsearch index to search.                      |
| **realert** | Controls repeated alerts for the same detection.                |
| **type**    | Defines how matching events trigger alerts.                     |

From an analyst point of view, this is useful because Sigma can be used as a starting point, then converted into tool-specific detection logic depending on the SIEM or alerting platform.

## ElastAlert Rule Configuration

After converting the Sigma rule, the generated ElastAlert rule needed small adjustments before execution.

The final rule was similar to:

```yaml
alert:
- debug
description: Sigma rule for sinkholed DNS queries.
filter:
- query_string:
    query: dns.resolved_ip:"0.0.0.0"
index: filebeat-*
name: dns_sinkhole
priority: 3
realert:
  minutes: 0
type: any
```

The most important parts of the rule were:

| Field          | Value                       |
| -------------- | --------------------------- |
| **Query**      | `dns.resolved_ip:"0.0.0.0"` |
| **Index**      | `filebeat-*`                |
| **Rule Name**  | `dns_sinkhole`              |
| **Alert Type** | `debug`                     |
| **Rule Type**  | `any`                       |

This configuration allowed ElastAlert to search the `filebeat-*` index and generate alerts for DNS events where the resolved IP address matched `0.0.0.0`.

> This helped me understand how threat intelligence and Sigma rules can be converted into operational detection rules that can generate alerts from log data.
